When to use this pattern

Use this pattern to tell users that data they expected to see cannot be displayed.

When not to use this pattern

Do not use this pattern to help users add missing information in a summary list. Instead, follow the show missing information in summary lists guidance.

How it works

There are 3 variations of this pattern:

  1. Data can be added in the service.
  2. Data needs adding outside of the service.
  3. Data can not be added.

Data can be added in the service

An example is data missing on a projects list page, that the user can add themselves.

In this case, make it clear that the user can add this information.

For example, an "Add this" link that takes the user to where they can add the information. Include hidden text so users know which piece of data the link relates to.

Data needs adding outside of the service

An example is missing data from GIAS (Get Information about Schools) that would normally appear in FIAT (Find information about academies).

A user may not be able to directly add the missing data in FIAT, but they may want to speak to a school and ask them to update GIAS.

In this case, make it clear that the data comes from another place.

For example, "Missing from GIAS (Get Information about Schools)".

If the user has permissions to add data to the external service, use a call-to-action link explaining where the user can add the data.

For example, "Add this to GIAS (Get Information about Schools)".

Data can not be added

An example is where an Ofsted inspection did not take place for a school, but users are expecting to see Ofsted data in your service.

Consider using "Not available" because it makes it clear that:

  • the data cannot be provided
  • your service is working as expected

Avoid words that suggest the user can add this data themselves, like "Empty".
