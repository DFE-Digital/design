When to use this component

Use this pattern to display read-only trust details. Provide another page to let users edit the details when required.

When not to use this component

Do not use this pattern if your design only has one trust details field. For example, a person's name.

How to use this component

If displaying trust details on a standalone page, use the summary list component.

If displaying trust details alongside other summary lists on a page, use the summary card design.

Whichever design you use, the trust details section should:

  • have more than one trust details field
  • provide a change link in the third column, if the fields are editable
  • point to one edit page, even if you have a change link for each field

It should not:

  • allow users to edit fields on the same page. For example, using AJAX-enabled forms. Take users to a separate edit page
  • provide a change link if users are not allowed to edit the information. This may be the case if a certain user group only needs to read it

Editing trust details

If users are allowed to edit the trust details, create an editing page and allow users to access it via the change link(s).

Retrieving trust details

Some users want to search for a trust using a trust reference number (TRN), trust ID, or trust name.

If this is the case, you can use a text input box and primary action button.

When users search for the trust, provide a page to allow users to confirm the retrieved information matches what they need.

When the user selects Yes, direct them to the relevant page in their user journey. If they select No, direct them back to the trust search page.

Entering trust details manually

There could be trust information that needs to be entered manually, rather than retrieved from a search. For example, the name and email address of a contact in the trust.

In this case, add extra rows in your summary list or card, and let users enter the information manually. See the contact details design pattern.

Research on this pattern

We have not done research on this design pattern, but the pattern is being used to meet these needs from our user needs database:

  • user need 1: When reaching out to a trust or academy, I need up-to-date details of people involved in the trust/academy so that I can get in touch with the right person
  • user need 24: When I need to contact a trust involved in a conversion or transfer project, I need to know the name and email address of the key trust contact, so that I can make contact with the correct person at the trust
